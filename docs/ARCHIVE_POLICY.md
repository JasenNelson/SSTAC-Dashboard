# Archive Policy

This policy keeps historical documents discoverable without polluting active documentation. It applies to markdown documents across the repository, not just `docs/`.

## Scope

- **Primary coverage:** markdown documents under `docs/`.
- **Also covered:** repo-root-level markdown (`*.md` at the repository root) and `.github/**/*.md`.
- **Archived under the same tree:** archives from any of the three locations above live under `docs/archive/YYYY-MM-DD_<topic>/`. Do not create parallel archive trees at the repo root or under `.github/`.
- **Out of scope (this policy):** markdown in other locations (e.g. `e2e/**/*.md`, `scripts/**/*.md`, `public/**/*.md`, tool-specific READMEs). Those are governed by the owning feature's conventions.
- **Never archived:** autogenerated artifacts (test reports, coverage, build output, `playwright-report/`, `coverage/`). Use `.gitignore` for generated files instead.

## Lifecycle states

Every markdown document covered by this policy is in exactly one lifecycle state at a time:

- **ACTIVE** — Load-bearing for current development. Must be kept correct; changes require review. The gate system may require this doc. Registered in the manifest with `lifecycle: AUTHORITATIVE`.
- **REFERENCE** — Helpful context that is not canonical. Must not claim global "current project status" or duplicate volatile metrics; must point at `docs/INDEX.md` and `docs/_meta/docs-manifest.json` → `facts`. Safe to update to keep accurate. Registered in the manifest with `lifecycle: REFERENCE`.
- **HISTORICAL-IN-PLACE** — A dated snapshot that stays in its original location because an older path is linked from external systems, the file is heavily cross-referenced, or archiving would cause more breakage than it resolves. Must carry a prominent stale-current-status banner (see §Stale-current-status banner). Registered in the manifest with `lifecycle: HISTORICAL` while still outside the `docs/archive/` tree.
- **ARCHIVE-CANDIDATE** — Flagged for a future archive move. The file stays in place until (a) a replacement doc exists, (b) the replacement is registered in the manifest and linked from an authoritative index, and (c) a small-batch archive plan is approved and reviewed by `/codex-review`.
- **REVIEW-NEEDED** — Lifecycle cannot be determined without a subject-matter read. Must not be archived from this state. Resolve to one of the four states above before taking action.

### Mapping to the manifest's `lifecycle` enum

`docs/_meta/docs-manifest.json` uses a three-value enum: `AUTHORITATIVE` | `REFERENCE` | `HISTORICAL`. Mapping:

| Policy state | Manifest `lifecycle` | Notes |
|---|---|---|
| ACTIVE | `AUTHORITATIVE` | |
| REFERENCE | `REFERENCE` | |
| HISTORICAL-IN-PLACE | `HISTORICAL` | File path is outside `docs/archive/`. |
| Archived (post-move) | `HISTORICAL` | File path is under `docs/archive/YYYY-MM-DD_<topic>/`. |
| ARCHIVE-CANDIDATE | (file keeps its current `lifecycle` until moved) | Classification lives in the archive/lifecycle audit, not the manifest. |
| REVIEW-NEEDED | (file keeps its current `lifecycle`) | Resolve before archiving. |

## Archive location and naming

- All archive folders live under `docs/archive/YYYY-MM-DD_<topic>/`.
- Archives originating at the repo root or under `.github/` move into the same `docs/archive/` tree; the archive README records the origin path.
- One topic per folder; do not mix unrelated docs in a single archive folder.
- Keep original filenames when archiving. Do not edit archived files.
- If a correction is needed after archiving, create a new dated archive folder rather than editing the existing one.

## Required archive README

Each archive folder must include a short `README.md` with:

- Purpose and scope of the archive.
- Creation date.
- Source/preceding document paths (pre-archive locations). For files originating outside `docs/`, record the original repo-relative path explicitly.
- Originating commit hash (if known).
- **Replacement proof** for every archived file (see next section).

## Source/replacement proof (required before any archive move)

A document may not be archived unless all three conditions hold:

1. There is a **named replacement** document — or an explicit "no replacement needed" rationale tied to a deprecation (e.g. the described feature is removed).
2. That replacement **exists in the working tree**, is registered in `docs/_meta/docs-manifest.json`, and is linked from `docs/INDEX.md` (or another authoritative index named in `docs/INDEX.md`).
3. `npm run docs:archive:investigate:all` passes on the current working tree (see §Investigation gate).

The archive folder's `README.md` must cite the replacement doc path and the commit/section that established the replacement. An archive move without a named replacement is rejected.

## Required manifest / INDEX updates

Every archive move requires, in the same commit or change set:

1. Update the moved file's `path` in `docs/_meta/docs-manifest.json` to the new archive path and set `lifecycle: "HISTORICAL"`. If the file was not previously registered (e.g. a root-level or `.github/` file), add a new entry.
2. Update or remove the `docs/INDEX.md` entry that referenced the pre-archive path. Replace it with a pointer to the archive folder under an appropriate "Archived" section.
3. Sweep for internal links that resolved to the pre-archive path. Either update them to the new path or confirm `docs:archive:investigate:all` flags them for follow-up.
4. Commit the move, the manifest update, and the INDEX update together.

## Facts vs facts_history

`docs/_meta/docs-manifest.json` separates live metrics from frozen snapshots:

- **`facts`** — live/current values only. Either auto-written by tooling (`facts.testing.*` via `scripts/verify/update-manifest-facts.mjs`) or human-curated canonical truth (`facts.grades.*` read by `scripts/analyze-grade.ps1`). Must carry provenance (`source`, `last_verified`).
- **`facts_history`** — frozen session-closeout / phase-complete snapshots. Read-only history. Must never be cited as "current".

When a live fact is superseded by a new session, move the prior entry into `facts_history` (optionally suffixing the key with the session date) rather than overwriting it in place. Do not introduce new top-level `facts.*` keys for historical work; add them to `facts_history`.

`last_verified` must be strict ISO `YYYY-MM-DD` in both buckets — no `(FINAL)` or session-label annotations.

## Stale-current-status banner (required for HISTORICAL-IN-PLACE)

When a document is kept in place but contains time-bound claims — "current grade", "phase complete", "production ready", rolling test counts, status dates — it must carry a banner stanza inside the first 20 lines:

```markdown
> **Lifecycle:** HISTORICAL-IN-PLACE (dated snapshot; do not treat as current truth).
> **Capture date:** YYYY-MM-DD.
> **For current status:** `docs/INDEX.md` (narrative) and `docs/_meta/docs-manifest.json` → `facts` (metrics).
> Claims below were true at capture and are not maintained afterward.
```

`docs/PROJECT_STATUS.md` and `docs/NEXT_STEPS.md` are canonical examples of the pattern.

## Small-batch rule

- Archive moves happen in small, named batches — **typically no more than six files per batch, never a broad sweep**.
- Each batch must be an approved plan tied to the most recent archive/lifecycle audit (currently `docs/_meta/DOCUMENTATION_ARCHIVE_LIFECYCLE_AUDIT_2026-04.md`) or a successor.
- A batch that mixes unrelated topics must be split.
- A single commit contains at most one batch plus its manifest/INDEX updates.

## Investigation gate

- `npm run docs:archive:investigate` — **narrow** scan (`docs/**` only). Preserves the existing gate semantics used by other docs tooling.
- `npm run docs:archive:investigate:all` — **wide** scan covering `docs/**`, repo-root `*.md`, and `.github/**/*.md`. Required before any archive batch moves files originating from any of those three locations.
- Do not archive when the investigation gate fails.
- Both gates run the same analysis (broken links, duplication clusters, inbound graph, TF-IDF unique terms) — only the root set differs.

## Review checkpoints

- Every archive batch must pass `/codex-review` on the policy/tooling diff produced alongside the current archive/lifecycle audit before the batch executes.
- `/codex-review` is a required checkpoint, not an optional one, for:
  - Changes to this policy.
  - Changes to `scripts/verify/docs-archive-investigation.mjs`.
  - Every archive batch execution.
  - Changes that touch `docs/_meta/docs-manifest.json` `facts` or `facts_history`.

## Prohibited content

- Do not archive autogenerated output such as `playwright-report/`, `coverage/`, or build artifacts.
- Use `.gitignore` for generated files instead.
