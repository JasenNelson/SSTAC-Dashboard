## The P2 (codex, verified by inspection)

The generator sets `candidate_group_id` to the 4-part join
`pathway__substance_key__input_key__jurisdiction` (generate-catalog-records.mjs ~line 291) but
disambiguates only `parameter_value_id` collisions with a slug suffix (~lines 343-352). So when two
distinct payloads share the same (source_id, substance_key, input_key, pathway, jurisdiction), they
get distinct ids but the SAME candidate_group_id -- colliding as indistinguishable duplicates.
25 such tuple groups exist in the integrated catalog.

Codex's required fix: add a semantic uniqueness guard on the tuple
(source_id, substance_key, input_key, pathway, jurisdiction); disambiguate legitimate conflicts by
basis or exclude+report bad ones; add a catalog test so the class cannot re-enter.

## The 25 groups -- CLASSIFIED (do not re-derive; just implement)

### CLASS 1 -- same-value pure duplicates (15 groups) -> COLLAPSE
The generator re-emitted a value that already exists in the BASE catalog under a different id slug
(same tuple, same normalized value). Examples (id pairs, all same value):
- benzo_a_pyrene rfd direct/food, sf direct/food: `pv-hc-bap-*` vs `pv-hc-benzo_a_pyrene-*`
- arsenic_inorganic sf direct/food: `pv-hc-arsenic-*` vs `pv-hc-arsenic_inorganic-*`
- lead rfd direct/food: `pv-hc-lead-*-risk-dose` vs `pv-hc-lead-*-rfd`
- cadmium rfd direct/food: `pv-hc-cadmium-*-tdi` vs `pv-hc-cadmium-*-rfd`
- copper rfd direct/food: `pv-hc-copper-*-tdi` vs `pv-hc-copper-*-rfd`
- vinyl_chloride rfd direct/food + rfc direct: `pv-iris-vinyl-chloride-*` vs `pv-iris-vinyl_chloride-*`
FIX: extend dedup to skip emitting a record when an existing OR already-generated record shares the
tuple AND the same normalized value. No information lost; no judgment. (Root cause: a later pass
re-derived a slightly different substance_key/id slug for a value already present.)

### CLASS 2 -- distinct-value legitimate candidates (8 groups) -> KEEP ALL, disambiguate by basis
Real source-backed alternatives the catalog is DESIGNED to hold (the iris-canonical test explicitly
allows multiple endpoint/medium values). The payload id slug already encodes the basis:
- IRIS benzo_a_pyrene oral RfD: neuro 3e-4 / repro 4e-4 / immune 2e-3 (slugs -neuro/-repro/-immune)
- IRIS cadmium oral RfD: water 5e-4 / food 1e-3 (-water/-food)
- HC methylmercury oral RfD: sensitive-subpop 2e-4 / general-adult 4.7e-4 (-sensitive/-adult)
- (+ each one's paired human-health-food row)
FIX: keep all; make candidate_group_id UNIQUE by appending a 5th `basis` segment
(`pathway__substance__input__jurisdiction__basis`) derived from the disambiguating slug. To minimize
churn across the thousands of single-candidate records, keep the 4-part form when a record is the
SOLE candidate for its tuple, and add the 5th `__<basis>` segment ONLY when 2+ distinct-value records
share a tuple. Comment the scheme in code; reflect it in the guard test.

### CLASS 3 -- dirty extractions (2 rows, currently inside class 2) -> EXCLUDE + ADJUDICATE report
Untrustworthy source extractions; must NOT ship:
- molybdenum P28 oral RfD value 0.02: source_excerpt has DOUBLED unit "mg/kg/d mg/kg/d" and a CAS
  bleed "(7439-98-7 91-20-3)" (the 2nd CAS is naphthalene's), id slug `...-soil-2`.
- dichloroethylene_1_2_trans P28 oral RfD value 0.006: source_excerpt has BLANK CAS "()", slug
  `...-soil-2`.
FIX: exclude from the shipped catalog; push an `ADJUDICATE: <id> <reason>` warning for HITL. Detection
heuristic (precise -- do not over-exclude): dirty if the excerpt unit token is duplicated
consecutively, OR the CAS in the excerpt is empty "()", OR contains two CAS numbers. Exclusion-with-
report only.

## JUDGMENT BOUNDARY (load-bearing)
AI must NOT choose which regulatory value wins among legitimate candidates. Class 1 collapse is safe
(identical value). Class 2 keeps ALL. Class 3 excludes only the DIRTY row and reports it. If a row
cannot be confidently classified as dirty by the heuristic, KEEP it as a class-2 candidate -- never
silently drop a regulatory value.
